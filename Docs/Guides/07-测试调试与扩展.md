# 07 - 测试、调试与扩展

## 测试分层

| 层级 | 位置 | 验证对象 |
|---|---|---|
| 纯 C# | `Tests/Core` | 标签、状态机、能力、战斗、AI、配置、存档规则 |
| EditMode | `Assets/_Project/Tests/EditMode` | 程序集边界、适配器和场景搭建结果 |
| PlayMode | `Assets/_Project/Tests/PlayMode` | 运行时生命周期、网络会话和交互表现 |
| 人工验收 | `Docs/Acceptance` | Unity 场景、构建、双开和操作体验 |

提交前按 [README](../../README.md) 的顺序运行核心规格、注释审计、中文文案审计及其审计脚本规格。`RunCoreTests.ps1` 会把 Core、Gameplay 和核心规格编译为独立程序，是验证领域层没有引入 Unity 依赖的快速方式。

## 调试入口

- Player 和 Enemy 自定义 Inspector：观察运行时状态，不应在 Inspector 中直接篡改规则。
- `Odyssey/配置`、`Odyssey/场景`、`Odyssey/联机`、`Odyssey/测试` 菜单：导入、搭建、联机和测试的可重复入口。
- F3：联机 RTT、连接人数和权威生命调试信息。
- 中文日志与测试报告：按代码规范保持可读；协议字段、文件路径和 API 名称保留英文。

## 新增功能的安全路径

1. 先确定规则属于 Core/Gameplay，还是 Unity/Network 适配；先写会失败的核心规格。
2. 通过配置、端口或事件连接层次，不让角色直接查找全局对象，也不让 UI 直接改规则。
3. 若功能改变网络可见状态，明确“谁提出请求、谁验证、谁写入、谁展示”。
4. 更新对应指南、验收文档和中文注释；运行最小相关测试后再跑门禁。

不要为了“以后可能用到”新增通用 EventBus、Manager、对象池或接口。当前项目偏好在出现真实的多个消费者、替换实现或测试替身时再抽象。

## 推荐源码练习

- 给现有 `Health` 规则新增一条可验证的拒绝条件，并补齐核心规格。
- 新增一个敌人配置字段，完成 CSV 导入、数据库、绑定、Inspector 观察和回归测试。
- 为一个新的网络动作设计请求序号、Host 验证和目标 Client 表现，再解释为什么不能由 Client 直接写生命。
