# 08 - 面试准备

面试回答应从“需求/约束 → 方案 → 取舍 → 验证”组织。以下问题都能回到源码和测试验证；不要把它们背成脱离项目的模板答案。

## 架构与工程化

### 为什么分成 Core、Gameplay、Unity、Network？

Core 保留可复用的无 Unity 规则，Gameplay 组合业务领域，Unity 承担引擎适配，Network 再把 NGO 接到运行时。结果是状态机、配置、存档和 AI 可以做纯 C# 测试，网络 API 不会污染角色规则。代价是需要显式 Installer/Adapter 代码；本项目接受这部分样板，以换取可测试性和依赖清晰度。

### 为什么不用单例？

应用级服务由 `OdysseyBootstrap` 创建并通过 `ApplicationContext`、`GameplaySceneInstaller` 显式装配。这样场景与测试不需要依赖隐式全局状态，场景重载时也能清楚地释放会话对象。不是完全禁止应用级唯一实例，而是禁止任意业务对象随时查找和修改它。

## 角色、战斗与 AI

### 为什么玩家要有两条状态轴？

移动与动作组合会指数增长。拆成 Locomotion 和 Action 后，移动状态不需要知道连击细节，动作状态也不会吞掉跳跃/落地逻辑。转移由纯策略和延迟提交状态机控制，便于单测，也避免一帧内重复 Enter/Exit。

### 伤害为何必须经过统一模型？

`DamageRequest → DamageResult → HealthChanged` 让无敌、死亡、UI、遭遇进度和网络表现都围绕一个事实入口工作。直接减字段会造成规则绕过、事件丢失和 Host/Client 双写。

### 行为树为什么是反应式？

敌人的感知事实每帧可能改变。反应式 Selector 从高优先级重新判断，使“发现玩家、受击、死亡”能立即抢占巡逻。它适合这个规模的可解释 AI；复杂长任务可再引入状态保持节点，而不是一开始过度设计。

## 联机

### Owner 权威移动和 Host 权威战斗是否矛盾？

不矛盾，它们分别服务于手感和公平性。Owner 快速同步移动/动画表现，Host 对可影响其他玩家的伤害、生命、怪物和关卡状态做最终判定。发生分歧时 Host 的修正结果优先。

### 如何防止 Client 重放或伪造攻击？

攻击请求携带递增序号，Host 检查序号、连击窗口、冷却、目标和距离，再用 Host 场景中的真实查询结算。请求被拒绝时只返回表现结果，不把 Client 的宣称当成伤害事实。它不是反作弊服务，但实现了作品集范围内正确的权威边界。

## 如何展示项目

1. 先演示单机闭环：移动、攻击、冲刺、AI、遭遇门和存档。
2. 再双开演示联机，打开 F3 并说明 Host/Client 分工。
3. 打开一条核心规格和对应实现，展示“规则先测、适配后接”的工作方式。
4. 最后说明范围边界：没有 Relay、匹配、完整 GAS 或确定性同步，并解释这是基于作品集时间和验证成本的主动取舍。

面试前按 [中文技术学习指南](README.md) 重走一次源码路径，并使用 [验收文档](../Acceptance/) 实机验证每个可演示结论。
